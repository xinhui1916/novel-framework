# Phase 3: Write — 逐章写作

## 核心原则（按优先级排序）

1. **灵感优先于计划。** 写到一半发现更好的方向 → 先按新方向写下去。写完后更新设定和大纲。
2. **每章独立上下文** — 每次只加载本章所需的信息，不拖拽全书的记忆。
3. **写完即更新** — 每章完成后更新注册表和追踪文件。

## 每章写作循环

### 1. 加载上下文
读取当前章节需要的信息：
- `bible/story-bible.md` — 核心设定摘要
- `bible/characters/_index.md` — 本章涉及的角色状态
- `plot/outline.md` — 本章的大纲描述
- `plot/plot-threads.md` — 本章涉及的伏笔/情节线
- `plot/timeline.md` — 确认时间线位置
- `chapters/_index.md` — 上一章结尾状态（避免衔接断裂）
- `progress.md` — 确认这是第几章

### 2. 前情提要
自动生成一句上一章的结尾状态（一句话），用于续写时快速进入状态：
> 例如：（上一章：主角发现了密室的暗门，但被神秘人抢先一步）

### 3. 节拍大纲
用 100-200 字列出本章节拍（分场景），用户确认后再开写。

### 4. 写作
- 按场景逐段输出
- 每章独立文件：`chapters/ch-NN-title-slug.md`
- 文件头用 YAML frontmatter 记录元数据
- 对话写作时参考角色声音库（如果项目有），每个角色说话方式要能区分
- **如果在写作中产生新想法**：先写下来，写完后标记"新设定待更新"，再去更新 bible
- 字数尽量达到目标，但如果"这里就该短"，可以调整目标

### 5. 自检（可选，灵感在线时可跳过）
- [ ] 字数达标（Python len 验证）
- [ ] 本章推进了至少一条情节线
- [ ] 结尾有钩子（除非是全书终章）
- [ ] 没有引入 bible 未定义的设定（如果有，标记待更新）
- [ ] POV 一致
- [ ] 对话角色区分度（不会所有角色说话一个调）

### 6. 更新注册表
- `chapters/_index.md` — 更新状态为 completed + 字数
- `plot/plot-threads.md` — 标记已触发的伏笔/情节
- `progress.md` — 更新进度

### 7. 审查暂停点（可跳过）
每 3 章建议暂停一次供用户审查，但灵感来了就直接说"继续"。

## 角色声音库（可选，推荐复杂作品开启）

多人一腔是 AI 写作最明显的弱点。如果项目评估时开启角色声音库，bible 中每个角色追加以下字段：

```
说话习惯：短句多/长句多/喜欢反问/爱用比喻
口头禅："不是我说" / "你懂吧" / 基本不用
语气词：很多（啊/呢/呀）/ 很少 / 没有
话题偏好：聊到钱就兴奋 / 回避感情话题
句式特征：喜欢排比 / 经常插入语 / 简洁
反应模式：先嘲讽再认真 / 先沉默再反问
```

写作时，每段对话前先看一眼说话角色的声音特征。不要求每句都完美符合，但角色间的对话要能区分。

## 去 AI 味（可选）
如果用户感觉 AI 味重，在每章写完后调去 AI 味流程：
- 检查禁用词（突兀的成语堆砌、模板化描述）
- 心理描写外化（不要"他感到"，用行动表现）
- 对话去腔调（每个角色说话方式是否区分度够——配合角色声音库检查）
- 节奏打碎（长短句交替，避免排比句堆砌）

## 字数约束（按项目画像）

字数模式在 Init 评估时确定：

- **宽松模式**：写好为止，字数自然产生。2000 字还是 5000 字都可以，不约束。
- **目标模式**：有字数目标但灵活。如果剧情精彩但字数不够 → "这里就该短"，调整目标即可。
- **硬约束**（网文连载）：稳定字数，达不到标记未完成。Python 统计实际字符数，不用估算。
